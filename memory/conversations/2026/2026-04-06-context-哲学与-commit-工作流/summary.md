# Context 哲学与 commit 工作流 摘要

## 这次对话解决了什么

- 读取并分析了用户提供的 3 段视频文案，确认其核心观点与当前正在搭建的 Git 记忆系统高度一致
- 将这些稳定观点提炼为新的主题主档案：
- `memory/domains/strategy/context_philosophy.md`
- 明确了这套系统的两层结构：
- 会话层：`memory/conversations/`
- 主题层：`memory/domains/`
- 明确了每次重要对话后不应只沉淀总结，还应判断是否需要更新固定主题
- 设计并落地了 `/commit` 工作流主题主档案：
- `memory/domains/collaboration/commit_workflow.md`
- 明确了 `/commit` 不是“保存所有聊天”，而是把本次对话中值得长期保留的新增内容沉淀回记忆系统
- 明确了连续工作、跨 agent 切换和新项目场景下的使用方式

## AI 后续应该记住什么

- 用户非常重视 context，不认为模型能力应该脱离 context 被单独评价
- 用户认为未来很多 AI 系统的核心价值，不在模型本身，而在于对个人或公司 context 的建模能力
- 用户希望 `/commit` 成为一种低摩擦、带有仪式感的收尾动作，而不是沉重的文档负担
- 用户不希望每次对话都机械 commit，而希望按“是否有长期增量”来判断
- 用户特别关注这套系统如何跨 agent、跨模型、跨设备继续使用，而不只是局限在当前 Codex 会话里

## 可复用约束 / 偏好 / 决策

- 默认把高价值长期信息沉淀到 `memory/domains/`，而不是只散落在会话层
- 每次重要对话结束后，优先判断：
- 是否形成稳定结论
- 是否有关键原话值得保留
- 是否需要更新已有主题
- 是否出现值得新增的固定主题
- `/commit` 适合在对话真正产生长期价值增量时触发，不按每天机械执行
