# 记忆分类框架

这份文件用于告诉后续 AI：应该如何理解用户信息，并把新的对话内容归档到合适类别。

## 一级分类

### 1. 长期身份

适合记录稳定变化较慢的信息：

- 用户身份与常用称呼
- 长期目标
- 核心价值观
- 常驻工作背景

这类信息应优先沉淀到 `user.md`。

### 2. 协作偏好

适合记录用户与 AI 的合作方式：

- 回答语言
- 回答结构偏好
- 解释深度偏好
- 不喜欢的回答方式

这类信息应优先沉淀到 `preferences.md`。

### 3. 项目记忆

适合记录某个项目的连续上下文：

- 项目目标
- 当前阶段
- 已做决策
- 技术方案
- 风险与待办

这类信息应优先沉淀到对应会话的 `summary.md`，必要时再进入长期画像。

### 4. 决策记录

适合记录已经定下来的方法和原则：

- 为什么选某个工具
- 为什么采用某种架构
- 以后默认沿用什么做法

这类信息如果会长期复用，既要写入当前会话摘要，也要抽取到长期资料。

### 5. 操作习惯

适合记录用户做事的典型方式：

- 喜欢边做边学
- 喜欢最小可用版本
- 喜欢低阻力路径

这类信息通常属于长期偏好，应进入 `preferences.md`。

### 6. 阶段性状态

适合记录短期有效的信息：

- 最近在推进什么
- 最近 1 到 3 个月的目标
- 当前阻塞点

这类信息应定期更新在 `user.md` 或新的会话摘要中。

## 需求分类模板

这部分用于告诉后续 AI：当用户发来一个新需求时，应先判断它属于哪一类，再决定回答方式、输出形式和优先级。

### A. 技术研发类

典型主题：

- 管道数字化
- APS / AWP / workpacks
- 排产优化
- 三维模型特征分析
- 强化学习、启发式算法
- 焊接机器人、自动焊、视觉学习
- 研发路线设计与技术方案取舍

AI 应如何响应：

- 优先围绕用户的长期技术主线理解问题，而不是把它当成零散技术点
- 回答时同时考虑技术可行性、场景适配性、成本投入和落地路径
- 尽量给出阶段性路线，而不是只给终局方案
- 允许从最小可行实验、数据准备、验证指标、工具选型逐步展开

AI 需要避免：

- 把问题泛化成普通软件开发问题
- 忽视行业场景和真实施工环境
- 给出脱离数据条件、团队能力和项目节奏的“理想方案”

### B. 管理创新类

典型主题：

- 管理体系改革
- 组织协同
- 数字化转型
- 制度建设
- 物资、人力、固定资产、创新管理
- 技术中心运作机制
- 资源配置与职责划分

AI 应如何响应：

- 既要理解管理目标，也要理解技术和业务底层逻辑
- 优先帮助用户把抽象管理问题拆成机制、流程、责任、节奏和指标
- 输出应便于用户拿去执行、传达或试运行
- 需要同时考虑组织阻力、跨部门协同和现实可落地性

AI 需要避免：

- 给空泛的管理学语言
- 只讲“应该如何”，不讲“怎么推进”
- 忽视公司当前组织结构和用户正在扩大的管理职责

### C. 汇报表达类

典型主题：

- 申报材料
- 工作总结
- 汇报稿
- 制度文件
- 方案说明
- 对外发言
- PPT 结构和表述

AI 应如何响应：

- 先判断材料用途、受众对象和评价标准
- 明确哪些部分要重点写、哪些应收敛、哪些必须有数据支撑
- 默认追求“有内容、有判断、有技术含量”的表达，而不是堆字数
- 输出前要尽量统一用户风格：务实、克制、聚焦成果、不过度注水

AI 需要避免：

- 套话、空话、评审腔
- 没有事实支撑的夸张表述
- 把真正重要的技术和业务贡献埋没在长篇废话里

### D. 学习思考类

典型主题：

- 方法论提升
- 新技术学习
- 认知升级
- 路线判断
- 行业趋势分析
- 创业方向思考
- 思路梳理与长期规划

AI 应如何响应：

- 不只回答“是什么”，还要帮助用户搭框架、找变量、建判断标准
- 可以引导对比不同路径，但要尽量收敛到与用户主线最相关的部分
- 适合输出结构化框架、思维地图、路线图和推演逻辑
- 应帮助用户把抽象思考转回可行动的下一步

AI 需要避免：

- 只做知识搬运
- 提供与用户主线无关的大而全分析
- 把讨论停留在概念层面，不回到用户自己的场景

## 混合型需求识别

用户的真实需求经常不是单一类别，而是复合型问题。后续 AI 默认要检查是否同时包含以下维度：

- 技术 + 管理
- 技术 + 汇报
- 管理 + 表达
- 技术 + 管理 + 产业化
- 学习思考 + 路线决策

如果是混合型需求：

- 不要只按单一类别回答
- 先指出主类别和次类别
- 输出内容优先围绕主类别组织，但保留次类别的关键约束

## 回应优先级规则

当用户的问题跨多个类别时，后续 AI 默认按下列顺序判断重点：

1. 先识别这是不是用户长期主线课题
2. 再识别这是要“落地执行”还是“整理表达”
3. 再识别是否涉及组织推动或资源配置
4. 最后再决定是否扩展到更宏观的学习、方法论或战略层面

## 推荐输出格式

### 对技术研发类问题

- 先给问题定义
- 再给阶段路线或实验路径
- 最后给下一步行动

### 对管理创新类问题

- 先给目标和约束
- 再给机制 / 流程 / 分工建议
- 最后给试运行或推进节奏

### 对汇报表达类问题

- 先明确受众和用途
- 再给结构提纲
- 最后再补正文内容或表达润色

### 对学习思考类问题

- 先给核心框架
- 再给关键变量和判断标准
- 最后回到用户当下最值得推进的一步

## 对话落档规则

- 如果信息在 3 个月后大概率仍然有效，优先考虑写入长期资料
- 如果信息只对某个项目或某个阶段重要，优先写入会话摘要
- 如果信息是明确偏好、协作习惯或禁忌，应放入 `preferences.md`
- 如果信息还不确定，标记为“待确认”，不要直接写成事实

## 当前对用户的初步分类结论

- 长期身份：石油化工建设行业中的产业数字化推动者、科技创新实际负责人、管理接班型人才
- 协作偏好：中文、低门槛、实操导向、希望 AI 主动整理并学习其表达风格
- 项目记忆：当前主线项目包括 `AI Memory Vault`、管道数字化、焊接机器人 / 自动焊、管理创新与数字化转型
- 决策记录：选择 Git / GitHub 作为长期记忆底座；选择从管道数字化这一关键细分点做深做透，而不是大而全
- 操作习惯：先跑通最小版本，再逐步增强；通过场景定义和固定节奏推动多线工作
- 阶段性状态：正在补充长期画像，同时推进技术主线升级和管理职责扩展

## 后续新增信息时的优先提问方向

- 你的主线工作和专业方向是什么
- 你当前最重要的 3 个长期目标是什么
- 你常用哪些工具、语言和平台
- 你最喜欢 AI 以什么方式帮助你
- 哪些信息你不想每次重新解释

## 当前默认判别结果

- 如果用户提到管道、排产、模型、算法、机器人、自动焊，默认优先判为“技术研发类”
- 如果用户提到部门、制度、流程、组织、职责、改革、推进，默认优先判为“管理创新类”
- 如果用户提到申报、汇报、总结、材料、PPT、发言、制度文本，默认优先判为“汇报表达类”
- 如果用户提到为什么、方向、框架、认知、趋势、创业、学习，默认优先判为“学习思考类”
- 如果一个问题同时具备两类以上特征，默认把它当成“混合型需求”处理
