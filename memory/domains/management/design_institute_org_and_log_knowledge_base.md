# 设计研究院组织口径与日志知识库底稿

这份主档案用于固定保存“设计研究院”相关的历史管理口径、当前钉钉组织口径，以及后续按部门沉淀日志知识库时应采用的结构化方案，避免把历史研发管理主体、当前行政组织树、日志抓取范围和知识库主题范围混为一谈。

## 0. 这次对话为什么重要

这次围绕设计研究院、钉钉部门与日志的对话，除了完成实际抓取任务，还承担了更深的记忆建设作用：

- 它让记忆系统更具体地理解用户当前面对的真实部门对象和管理范围。
- 它把部门 ID、员工 UserID、成员归属和日志结果转成了可核验的组织证据。
- 它为后续做部门知识库和数字化分析建立了事实底座。
- 它也是 AI 学习用户如何把“任务、证据、知识沉淀、风格校准”放在同一条主线里的典型样本。

## 1. 历史管理口径

根据仓库内既有研发管理资料，用户过去一段时间的研发管理与奖励机制，存在明确的组织口径演化：

- 2023 年至 2024 年，季度研发管理计划和奖励文件主要以“设计研究院”名义发布。
- 2025 年起，相关资料逐步切换为“技术中心”名义，说明研发管理工作从部门型创新管理，过渡到技术中心实体化运行。
- 因此，“设计研究院”在历史资料中首先是一个研发管理和课题组织口径，而不应直接等同于当前钉钉中的某一个单独部门 ID。

这一点对于后续整理日志知识库非常重要：

- 历史研发管理口径：回答“过去哪些人、哪些课题、哪些奖励机制被纳入同一个管理体系”。
- 当前钉钉组织口径：回答“今天日志接口到底能按哪个部门树抓数据”。
- 知识库口径：回答“我们最终想沉淀的是哪一组人、哪一条业务线、哪一种管理与研发协作实践”。

## 4. 后续知识库应采用的四层结构

从全局角度，后续不建议把“抓到的 Markdown 日志”直接当成最终知识库，而应拆成四层：

### A. 组织主数据层

记录相对稳定但可能随时间变化的组织主数据：

- 部门名称
- 部门 ID
- 父部门 ID
- 员工姓名
- 员工 UserID
- 员工挂接的多个部门 ID
- 是否在指定月份有日志

这一层适合放在：

- `memory/imports/组织架构/`

它是自动化处理和后续更新的底表，不应直接承担解释性结论。

### B. 原始日志证据层

保存从钉钉导出的原始 Markdown 日志：

- 按人分文件
- 按月份分目录
- 尽量保持原始字段顺序

这一层适合继续放在：

- `outputs/dingtalk_logs_*/`

它是事实证据层，后续摘要、专题沉淀和考核判断都应可追溯回这里。

### C. 部门知识整理层

在原始日志之上，逐步形成按部门视角的主题汇总，例如：

- 月度重点事项
- 人员分工与协作关系
- 项目推进节奏
- 常见问题与阻塞
- 材料代码、预制深度、IDF/PCF、氚云、Three.js 等专题脉络

这一层更适合逐步沉淀到：

- `memory/domains/management/`
- `memory/domains/technology/`

它属于可复用结论层，而不是原始导出层。

### D. 管理评价与激励层

如果后续要把日志用于研发管理、奖励或部门评价，不建议直接从单篇日志做判断，而应叠加：

- 课题计划
- 会议纪要
- 奖励办法
- 氚云任务痕迹
- 知识库协作痕迹
- 项目交付结果

这与仓库中既有研发管理逻辑是一致的：过程证据和知识沉淀，本来就是考核体系的一部分。

## 5. 推荐的后续推进顺序

建议按下面顺序推进，而不是一次性把所有日志直接汇总成一份大文档：

1. 固化组织主数据
2. 按月份持续抓取日志
3. 先做部门月报式摘要，再做专题知识库
4. 再把专题结论反哺到研发管理与考核体系

更具体地说：

1. 每次新增一个月份日志时，先更新成员主数据表中的“是否有日志”和时间范围字段。
2. 对每个月的日志做一次“部门级月度摘要”，关注人力投入、项目主线、问题与风险。
3. 当某个主题跨多月稳定出现时，再单独沉淀为专题知识，例如：
   - 材料代码系统
   - 预制深度审核平台
   - IDF/PCF 数据处理
   - 氚云数据治理
4. 当日志知识与季度计划、奖励机制、会议纪要能够互相印证时，再更新管理主档案。

## 6. 当前最重要的结论

- “设计研究院”在仓库历史中主要是研发管理口径，不应直接等同于当前钉钉某一个行政部门。
- 当前已打通的日志抓取主范围，是 `917593636` `设计研究院（马来西亚）`。
- 这个范围足以作为第一版部门日志知识库的可信基础。
- 后续如果要扩大为更完整的“设计研究院架构”，应采用“历史管理口径 + 当前部门树 + 实际日志可得性”三者叠加的方式，而不是只看部门名称。
