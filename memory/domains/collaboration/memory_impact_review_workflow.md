# 记忆影响分层工作流

这份文件用于规定：每次重要对话在 `/commit` 前后，后续 AI 不应只新增文件，还应明确判断这次对话到底影响了记忆系统的哪一层，哪些是主影响，哪些是次影响，哪些只是局部证据。

## 1. 为什么要有这层

如果每次 commit 只是：

- 新增几个文件
- 更新几个主题
- 推送到 GitHub

但没有回答“这次到底补了哪层记忆”，后续就会出现：

- 文件越来越多
- 主题越来越散
- 新模型不知道这次对话的主价值在哪里
- 资料能找到，但学习路径不清晰

因此，后续 AI 在处理重要对话时，必须增加一层“记忆影响分层复盘”。

## 2. 默认要回答的 5 个问题

每次重要对话结束后，后续 AI 默认回答：

1. 这次对话的主影响层是什么？
2. 这次对话的次影响层是什么？
3. 这次新增了哪些证据层材料？
4. 哪些内容不应升级为主记忆？
5. 哪些主档案因此需要更新？

## 3. 推荐输出模板

后续 AI 默认应尽量产出一个简短的“记忆影响摘要”，格式如下：

- 主影响层：
- 次影响层：
- 证据层新增：
- 不应升级为主记忆的内容：
- 需要更新的主档案：

必要时可以再补：

- 这次对话在整体记忆中的位置：
- 是否形成了新的长期主题：
- 是否只是某个已有主题的补证据：

## 4. 当前推荐的影响层

后续 AI 可优先从下面这些层中判断：

- 长期身份
- 协作偏好
- 技术课题
- 管理机制
- 表达风格
- 证据导入
- 人脉网络

注意：

- “人脉网络”是上位层，尤其适用于部门、成员、组织边界、上下级关系、客户和协作对象等主题。
- “证据导入”通常不是主影响层，而更常是次影响层。
- “表达风格”不应被工具路径、接口细节或系统 ID 污染。

## 5. 这次对话提供的校正规则

本次围绕设计研究院、钉钉、部门日志和记忆分层的连续对话，提供了一条非常重要的校正规则：

- 不要把“本次为了完成任务所使用的工具路径”，误写成长期主风格或主记忆。

例如：

- 部门 ID
- 人员 UserID
- 接口调用方式
- 脚本实现方式

这些都更适合作为证据层或工具层，而不是用户长期画像或风格本身。

相反，更接近主记忆的是：

- 用户当前真实管理和关注的部门对象是谁
- 这类对象在用户的人脉网络中处于什么位置
- 用户希望 AI 如何从局部任务中抽取整体有价值的理解

## 6. 与 `/commit` 的关系

后续 `/commit` 的默认顺序应是：

1. 先加载整体记忆
2. 再判断当前主题段增量
3. 再做记忆影响分层复盘
4. 再决定更新哪些主档案
5. 最后生成 `active_context.md` 并提交推送

也就是说：

- 记忆影响复盘不是附加动作
- 它应成为 `/commit` 的一部分

## 7. 当前推荐的落盘位置

不同层级的结果，建议这样落盘：

- 固定规则：
  写入本文件和 `/commit` 工作流
- 单次对话的影响摘要：
  写入该次会话目录中的 `summary.md` 或单独 `memory_impact.md`
- 稳定结论：
  再根据影响层分别更新 `user.md`、`preferences.md`、主题主档案等
