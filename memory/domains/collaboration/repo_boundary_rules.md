# memory 仓库与项目仓库的边界规则

## 这份规则解决什么问题

当用户在不同对话中持续上传附件、推进多个项目、并且有时需要同步到不同 Git 仓库时，必须明确：

- 什么内容应该进入主 memory 仓库
- 什么内容应该进入项目仓库
- 什么内容可以两边都出现，但承担不同角色
- 如何避免重复堆积
- 如何避免只存一边导致后续断链

这份规则的目标不是增加归档负担，而是保证长期一致性。

## 与现有规则的关系

这份规则是对以下既有原则的细化，不是替代：

- `README.md` 中关于 `memory/imports/` 是证据来源层、`memory/domains/` 是稳定主档案层的原则
- `commit_workflow.md` 中关于“一次主题段落形成稳定增量再 /commit”的原则
- `domain_registry.md` 中关于只有跨多次对话仍稳定有效的内容才进入 domain 主档案的原则

换句话说：

- 主 memory 仓库仍然是长期 context 中枢
- 项目仓库不是 memory 仓库的竞争者，而是原始材料和执行现场的承载体

## 一级原则

### 1. 主 memory 仓库保存什么

主 memory 仓库优先保存：

- 对话摘要
- 稳定结论
- 关键证据摘录
- open questions
- domain 主档案
- 少量高复用、小体积、跨项目仍重要的附件
- 指向原始资料所在位置的索引信息

主 memory 仓库的角色是：

- 用户长期记忆中枢
- 跨模型、跨 agent、跨电脑可复用的 context 中心
- 长期稳定认知和方法论的沉淀层

### 2. 项目仓库保存什么

项目仓库优先保存：

- 原始 Word / Excel / PDF / PPT / 图片 / 压缩包
- 项目专属过程文档
- 项目内部版本演进文件
- 图纸、表格、报告原件
- 与某一项目强绑定、复用性弱但资料价值高的附件

项目仓库的角色是：

- 原始材料承载层
- 项目执行现场
- 某个专题或项目的完整文档现场

### 3. 两边都可以记录什么

当某份内容既有项目原件价值，又有长期记忆价值时：

- 项目仓库存原件
- memory 仓库存摘要、索引、证据摘录、结论和指针

这不是重复，而是分层。

## 二级规则

### 1. 原件优先归项目仓库

如果一份文件满足以下任一条件，优先归入项目仓库：

- 体积较大
- 版本较多
- 只服务于单个项目
- 原始格式价值高于摘要价值
- 后续还需要多人围绕它反复编辑

### 2. memory 仓库只保留“沉淀后的稳定层”

如果一份文件的主要价值在于：

- 提炼后的判断
- 可复用的方法
- 对用户长期工作方式的影响
- 对后续模型切换时的 context 价值

则应进入 memory 仓库的 conversation 或 domain 层。

### 3. `memory/imports/` 的定位不变

`memory/imports/` 仍然只是：

- 抽取后的文本层
- 证据来源层
- 便于检索与再分析的中间层

它不是：

- 长期主档案层
- 所有原始附件的永久存放层

如果某批材料已经稳定存在于项目仓库，且无需跨项目长期保留原件，则 memory 仓库不必重复保存完整原始附件。

## 多电脑协作规则

### 1. 远端 GitHub 仓库才是主库

对于 `ai-memory-vault`：

- 远端 GitHub 仓库是主库
- 本地 clone 只是某台电脑上的工作副本
- 换电脑不要求必须已有 clone 才能继续协作
- 只要能访问同一个远端仓库，就可以继续同步

### 2. 本地 clone 不是必需条件

用户不需要在每台电脑上都 clone memory 仓库，除非：

- 需要本地全文搜索
- 需要脚本批量处理
- 需要在本地编辑器中集中整理
- 需要执行本地 Git 工作流

否则，远端连接器工作流就足够支撑跨电脑使用。

### 3. 删除的判断标准

- 只在某台电脑本地删除，但未 push：只影响该电脑本地副本
- 删除后 commit 并 push：远端被修改，其他电脑 pull 后也会同步该删除结果

所以真正的全局变化，以远端仓库状态为准。

## 多仓库协作规则

### 1. 主 memory 仓库是“总记忆”

`ai-memory-vault` 应长期承担：

- 总索引
- 总结论
- 总证据线索
- 长期 domain 方法论

### 2. 项目仓库是“项目现场”

不同项目可拥有自己的仓库，保存：

- 原件
- 项目过程
- 附件全集
- 项目专属版本演进

### 3. 避免重复的标准方法

避免重复的核心不是“禁止两边都提到同一个文件”，而是：

- 原件只保存在一个主位置
- memory 仓库只保存摘要与索引
- 对话总结里清楚写出原件位置

推荐写法：

- 在 `evidence.md` 中明确：原件在哪个项目仓库、路径是什么、关键判断来自哪里
- 在 `summary.md` 中只保留提炼后的稳定结论
- 在 `domain_updates.md` 中判断这次内容是否值得更新长期主档案

## 每次对话后的归档判断

每次新对话结束后，优先判断以下 3 个问题：

1. 这次新增的是原始项目材料，还是长期稳定记忆？
2. 这次内容的原件应归哪个项目仓库？
3. 这次对话里形成了哪些值得进入 memory 仓库的稳定结论？

如果答案是：

- 原始材料：进项目仓库
- 稳定结论：进 memory 仓库
- 两边相关：项目仓库存原件，memory 仓库存摘要、证据和索引

## 实际执行口径

后续默认采用以下口径：

- `ai-memory-vault` 放“你的大脑”
- 项目仓库放“项目的身体”
- memory 不替代项目仓库
- 项目仓库不替代 memory

## 对后续 AI 的要求

后续 AI 在建议归档位置时，应先检查：

- 是否符合 `README.md` 对 imports / domains / conversations 的既有定位
- 是否符合 `commit_workflow.md` 对 /commit 的既有规则
- 是否符合本文件对主 memory 仓库与项目仓库边界的规定

如发生冲突，优先原则是：

1. 不破坏主 memory 仓库的长期可读性
2. 不把项目原件无差别搬入 memory 仓库
3. 不让稳定结论只留在项目仓库而不进入 memory 层
4. 不因临时方便而破坏长期一致性
