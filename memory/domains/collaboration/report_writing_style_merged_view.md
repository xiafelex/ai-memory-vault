# Report Writing Style Merged View

## 1. Purpose

This file consolidates the current report-writing rules that are distributed across multiple collaboration style files and recent content-engineering adjustments.

It is intended as a practical merged view for future use.

## 2. Source Layers

The merged view draws from four sources:

1. existing writing style rules
2. traceable response rules
3. style calibration rules
4. recent user corrections from report-writing work

## 3. Stable Shared Rules

### 3.1 Expression goal comes before structure

Before writing, define:

- what this document is trying to achieve
- who it is written for
- what the reader needs to understand quickly

The structure should serve that goal.

### 3.2 Results and judgments should appear early

Report material should lead with:

- result
- conclusion
- key system
- key data
- major issue

Do not make the reader wait through long setup text.

### 3.3 Structure must be stable

The hierarchy, chapter boundaries, term usage, and data presentation should be consistent.

The reader should be able to tell what each section is for.

### 3.4 Evidence must be traceable

Important judgments should be explainable by source, scenario, and example.

If evidence is incomplete, the text should say so through moderated judgment wording.

### 3.5 User corrections are high-weight evidence

Explicit user revisions, especially repeated ones, should become stable style rules.

Rejected patterns should not be reused casually.

## 4. What Changed in Recent Report Writing

Recent revisions clarified several rules that were already implicit but are now stable enough to use by default.

### 4.1 From analysis-led to conclusion-led

Earlier drafts often showed the thinking path first.

The preferred style now states the recommendation or judgment first, then follows with support.

### 4.2 From material accumulation to topic control

Past-project records, raw tables, and supporting files are no longer treated as the main story.

They are evidence layers that support the current topic.

### 4.3 From mixed chapters to stable chapter roles

The chapter should have a clear function and keep that function.

For example:

- design chapter handles responsibility and scope
- procurement chapter handles purchasing boundary and supplier logic
- logistics chapter handles shipment and delivery control
- schedule chapter handles duration, critical path, and delay factors

### 4.4 From abstract phrases to project language

Expressions that sound analytical but are hard to understand should be translated into concrete project wording.

### 4.5 From broad claims to calibrated claims

If facts do not fully support a strong conclusion, the sentence should be softened rather than overstated.

## 5. Practical Rules for Report Text

### 5.1 Default writing order

Use:

1. conclusion
2. support
3. boundary
4. impact
5. recommendation

### 5.2 Data rule

Use numbers only when they help the reader understand a judgment.

Default downgrade items such as:

- unit numbers
- drawing counts
- prices
- long raw date chains

unless they directly support the point.

### 5.3 Table rule

If the content is parallel, list-like, or comparative, prefer a table or split-line format.

Do not bury equivalent items in a dense paragraph.

### 5.4 Language rule

Reduce:

- 从资料看
- 综合分析
- 可以看出
- 不是……而是……

Prefer direct statements.

### 5.5 Vocabulary rule

If a phrase is easy for insiders but hard for general project readers, replace it with a more concrete project expression.

## 6. General Requirements That Now Look Stable

The following rules now appear stable enough to treat as default report-writing requirements:

- conclusion first
- support second
- the current topic stays as the main line
- past cases stay in the evidence layer
- each chapter has a fixed job
- use tables for parallel information
- avoid low-value detail unless it supports the point
- avoid internal jargon when a plain project phrase is available
- calibrate judgment strength to evidence strength
- do not recycle rejected wording patterns

## 7. What Still Needs Ongoing Optimization

These areas should continue to be calibrated as more materials are produced:

- how much data is enough before a paragraph becomes overloaded
- when a chapter should be shortened into a management-report version
- how far to compress wording for PPT versus text reports
- which phrases are acceptable industry shorthand and which are still too internal

## 8. One-Sentence Default Rule

For most report-style writing, default to:

Clear conclusion first, evidence-backed support, stable structure, direct project language, useful data only, and judgment strength matched to proof strength.
