# Report Writing Style

## 1. Purpose

This file defines the default writing style for most report-oriented text output.

It is not limited to one project. It applies across:

- project planning reports
- phase reports
- management briefings
-专题分析
- comparison papers
-复盘总结
- PPT page copy

The goal is not to make the text look polished in a generic way. The goal is to make the text readable, usable, traceable, and immediately actionable in a reporting context.

## 2. Core Principles

### 2.1 Conclusion first

Report text should state the conclusion first, then give support.

The reader should not have to follow the reasoning process before understanding the point.

### 2.2 The main line must be clear

Each section should answer one primary question.

Do not mix:

- conclusion
- source review
- internal reasoning
- side background

in the same paragraph unless there is a strong need.

### 2.3 Evidence supports the conclusion

Historical projects, source materials, tables, schedules, contracts, and raw records are evidence layers by default.

They support the current topic. They do not become the narrative main line by themselves.

### 2.4 Practical language over analytical language

The writing should sound like reporting work, not showing analysis ability.

Use clear project language that can be understood by practitioners and non-direct participants.

### 2.5 Useful data only

Numbers should be kept only when they support a judgment.

Do not pile up:

- unit numbers
- drawing counts
- prices
- historical detail

unless they directly help the current conclusion.

### 2.6 Moderate judgment strength

If evidence is incomplete, do not write as if the conclusion is fully proven.

Prefer:

- 建议
- 宜
- 可按
- 可优先
- 当前判断
- 初步建议

over absolute statements.

## 3. Structural Rules

### 3.1 Default section order

For most report-style sections, use this order:

1. current conclusion or recommendation
2. key facts and key data
3. necessary boundary clarification
4. issue or impact
5. recommendation or implication

### 3.2 Parallel information handling

Use tables or split lines for parallel items.

Use paragraphs for:

- judgment
- explanation
- implication

Do not overload paragraphs with long lists of equivalent items.

### 3.3 One section, one job

Each chapter should have a stable role.

Examples:

- design chapter explains responsibility and scope
- procurement chapter explains boundary, sourcing, and supplier logic
- logistics chapter explains transport organization and delivery risk
- schedule chapter explains duration, critical path, delay factors, and control logic

### 3.4 Fixed placement for cross-cutting content

Cross-chapter topics should have fixed placement and should not be repeated randomly.

Examples:

- BIM and digital planning
- domestic prefabrication plus overseas installation
- labor strategy
- local approval preparation

These should appear once in the chapter where they are most relevant.

## 4. Language Rules

### 4.1 Avoid analysis-style openings

Reduce or avoid:

- 从资料看
- 综合分析
- 可以看出
- 执行经验表明

unless the wording is truly necessary.

### 4.2 Avoid rhetorical skeletons

Do not overuse:

- 不是……而是……
- 既……又……还……

as the main structure of the text.

State the definition, boundary, or recommendation directly.

### 4.3 Translate jargon into project language

Avoid internal analytical phrases or abstract terms that make the audience stop and decode the sentence.

Prefer concrete project wording.

For example:

- replace 外部接口 with 厂外管线、码头连接、外部公用工程接入、第三方协调 when those are the real meanings
- replace 收口 with 完成、接收、达到节点
- replace 打穿关键路径 with 开始影响项目总体里程碑

### 4.4 Professional but plain

The tone should be professional, stable, and understandable.

The text should not feel decorative, slogan-like, or overly polished.

## 5. Data and Evidence Rules

### 5.1 Every important judgment should answer "based on what"

For each important conclusion, the writer should be able to answer:

- what fact supports it
- what number supports it
- what case supports it

### 5.2 Keep only high-value numbers

High-value numbers usually include:

- duration
- delay
- completion time
- delivery time
- approval time
- labor ratio
- prefabrication ratio
- critical path related milestones

Low-value numbers should be downgraded unless directly useful.

### 5.3 Past project evidence stays subordinate

Past-project numbers should support the current recommendation, not take over the writing.

The main sentence should still be about the current project or current topic.

## 6. Calibration Rules

### 6.1 User corrections outweigh smooth draft language

If the user explicitly rewrites or rejects an expression, that correction is higher-value evidence than an apparently fluent initial draft.

### 6.2 Rejected patterns should not be recycled

If a wording pattern has been clearly rejected, do not bring it back in a slightly different form.

### 6.3 Do not confuse genre requirements with personal style

Some structure belongs to the document type. Some belongs to the user's stable preference.

Keep those two layers separate when updating style memory.

### 6.4 The user usually wants "more accurate and more usable", not "more literary"

When revising report text, prefer:

- clearer focus
- more accurate scope
- better judgment calibration
- stronger evidence support

over prettier but less usable language.

## 7. Default Output Pattern

For most report-oriented text, the default style can be summarized as:

Conclusion first, evidence after; clear main line, direct language, useful data, moderate judgment, and stable chapter boundaries.

## 8. Common Red Flags

The following are warning signs that a report draft is drifting away from the preferred style:

- too many analysis lead-ins
- too much historical detail in the main line
- vague strategic wording with little evidence
- repeated content across chapters
- too many abstract or insider terms
- too much emphasis on material volume rather than decision value
- unsupported strong conclusions

## 9. Working Shortcut

When producing report text, start with these checks:

1. What is the conclusion of this chapter
2. What is the minimum evidence needed
3. What details can be downgraded
4. Is the wording understandable without internal context
5. Is this paragraph reporting, or just thinking out loud
