# 多设备知识库工作流

## 目标

这份文件用于规定：当用户在不同电脑、不同 agent、不同线程中上传资料、讨论问题和沉淀结论时，知识库应如何保持统一。

核心原则不是让所有设备都保存完全相同的原始文件，而是让不同设备都尽量把经过筛选的结构化结论、证据摘录和主题更新汇总回同一个远程主记忆仓库。

## 为什么需要这套工作流

如果完全依赖本地仓库，会出现以下问题：

- 不同电脑上的附件不完整
- 临时导入、临时分析过的文件容易散落在多台机器
- 多设备同时工作时，容易形成多个半完整版本

如果完全依赖远程仓库，也会有问题：

- 不是所有原始附件都适合进入 Git 仓库
- 很多文件较大、较杂，或只是阶段性参考
- 过程性输出和临时缓存不应作为长期主记忆的一部分

因此，更合理的方式是：

- 本地负责处理
- 远程负责沉淀

## 双层结构

### 1. 远程主记忆层

远程主记忆层通常对应 GitHub 上的主仓库。这里应优先保存：

- 主题主档案
- 会话摘要
- 证据摘录
- 关键资料的文本抽取结果
- 重要原文的结构化摘要
- 申请表、成果报告、课题综述等高价值底稿

这一层的特点是：

- 跨设备
- 跨 agent
- 可追溯
- 应作为长期主版本

### 2. 本地工作层

本地工作层用于不同设备上的处理和过渡。这里可以保留：

- 临时附件
- 大文件原件
- 待整理会议纪要
- PPT 输出
- 依赖目录
- 脚本和缓存

这一层的特点是：

- 主要服务于处理和试验
- 不要求全量同步
- 只把真正值得沉淀的内容送回远程主记忆层

## 不同设备上传不同附件时的处理原则

不同设备可以处理不同原始附件，但不应要求所有设备都拥有完全一致的原件集合。

真正需要统一的是：

- 结构化结论
- 关键证据摘录
- 主题更新
- 可以复用的工作底稿

例如：

- A 电脑上传并分析了文件 X
- B 电脑上传并分析了文件 Y

这时不要求：

- A 电脑必须马上拥有 Y 的原始文件
- B 电脑必须马上拥有 X 的原始文件

但应尽量保证：

- X 的结论、证据和主题更新进入远程主记忆层
- Y 的结论、证据和主题更新也进入远程主记忆层

## 附件分层规则

### A 类：核心长期资料

典型包括：

- 立项报告
- 成果评价申请表
- 重要会议纪要
- 红头文件
- 核心论文
- 核心项目总结

处理原则：

- 原件可以保留
- 文本抽取结果应尽量保留
- 建议至少将抽取文本、结构化摘要和主题更新进入主仓库

### B 类：阶段性参考资料

典型包括：

- 临时比较材料
- 某次汇报参考稿
- 某个阶段的外部资料

处理原则：

- 原件不一定进入 Git
- 但重要结论和关键证据可以进入主仓库

### C 类：过程性大文件

典型包括：

- PPT 输出
- 渲染文件
- 依赖目录
- 中间缓存

处理原则：

- 保留在本地工作层
- 不进入主记忆仓库

## 多设备场景下的推荐工作流

在任意设备上，推荐流程如下：

1. 上传资料
2. 读取和分析资料
3. 生成结构化结论、证据摘录和主题更新建议
4. 将值得沉淀的内容写回远程主记忆层
5. 原始大附件留在本地，除非它属于核心长期资料

重点不是“所有设备完全一致”，而是“所有高价值知识尽量回流同一个主仓库”。

## 推送与上传拆分规则

为了降低网络不稳定、大文件超时和远程更新失败的风险，推荐将推送拆成“小文件优先、大文件后置”的方式。

### 基本原则

- 优先推送结构化结论、抽取文本、主题更新和脚本等小文件
- 原始 PDF、会议纪要原件、其他大附件尽量后置
- 单次上传建议尽量控制在约 `100MB` 以内
- 如果单个文件本身超过 `100MB`，则单独处理，不与其他文件混推
- 同一批待上传文件，建议按体积从小到大处理

### 这样做的原因

- 可以优先保证远程主记忆层尽快获得结构化更新
- 可以减少一次性推送大量二进制文件导致的网络超时
- 即使大文件后续未及时上传，主知识库中的结论、证据和主题更新仍然可用

### 拆分时的优先级建议

1. `memory/domains/`、`summary.md`、`evidence.md`、`active_context.md`
2. `memory/imports/` 中的抽取文本和结构化摘要
3. 小型脚本、配置文件、依赖说明
4. 原始 PDF、会议纪要原件、其他大附件

## 拆分上传时如何避免 memory 记录分散

推送可以分批，但知识主题不应因此被机械拆碎。

换句话说：

- `git commit` 的拆分可以服务于网络和文件体积管理
- 但 `memory` 的主题记录仍应按“一个主题段、一条记录”的原则组织

例如：

- 如果同一主题段同时产生了主题更新、小型脚本和大 PDF 原件
- 可以先提交并推送主题更新和脚本
- 再在后续单独处理 PDF 原件

此时：

- `memory/conversations/` 中仍只需要保留一条主题记录
- 不需要因为文件分两批上传，就人为拆成两条会话记忆

因此，后续 AI 在处理分批推送时应区分两件事：

1. 这是文件传输拆分
2. 这是否是新的知识主题

只有第二种情况才应新建会话记录。

## 与连接器和远程调用的关系

GitHub 连接器或远程仓库更适合处理：

- 主记忆同步
- 仓库状态
- 结构化沉淀

本地设备更适合处理：

- PDF、Word、Excel、会议纪要等原始附件
- 文本抽取
- 临时分析
- 输出文件生成

因此，连接器不应被视为本地附件处理的替代，而应被视为主记忆层同步和远程状态管理的工具。

## 后续 AI 的默认行为

当用户在不同设备上提供新资料时，后续 AI 默认应优先判断：

1. 这是原始附件、结构化底稿，还是长期主档案内容？
2. 应留在本地工作层，还是写回远程主记忆层？
3. 是否值得更新 `memory/domains/`、`summary.md`、`evidence.md` 或 `active_context.md`？

默认目标不是收集所有文件，而是降低未来跨设备协作和跨 agent 协作的理解成本。

## 一句话原则

不同设备可以处理不同原始附件，但必须尽量把经过筛选的结构化结论、证据摘录和主题更新汇总回同一个远程主记忆仓库。
