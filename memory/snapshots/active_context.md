## 记忆分类框架

这份文件用于告诉后续 AI：应该如何理解用户信息，并把新的对话内容归档到合适类别。

## 一级分类

### 1. 长期身份

适合记录稳定变化较慢的信息：

- 用户身份与常用称呼
- 长期目标
- 核心价值观
- 常驻工作背景

这类信息应优先沉淀到 `user.md`。

### 2. 协作偏好

适合记录用户与 AI 的合作方式：

- 回答语言
- 回答结构偏好
- 解释深度偏好
- 不喜欢的回答方式

这类信息应优先沉淀到 `preferences.md`。

### 3. 项目记忆

适合记录某个项目的连续上下文：

- 项目目标
- 当前阶段
- 已做决策
- 技术方案
- 风险与待办

这类信息应优先沉淀到对应会话的 `summary.md`，必要时再进入长期画像。

### 4. 决策记录

适合记录已经定下来的方法和原则：

- 为什么选某个工具
- 为什么采用某种架构
- 以后默认沿用什么做法

这类信息如果会长期复用，既要写入当前会话摘要，也要抽取到长期资料。

### 5. 操作习惯

适合记录用户做事的典型方式：

- 喜欢边做边学
- 喜欢最小可用版本
- 喜欢低阻力路径

这类信息通常属于长期偏好，应进入 `preferences.md`。

### 6. 阶段性状态

适合记录短期有效的信息：

- 最近在推进什么
- 最近 1 到 3 个月的目标
- 当前阻塞点

这类信息应定期更新在 `user.md` 或新的会话摘要中。

## 需求分类模板

这部分用于告诉后续 AI：当用户发来一个新需求时，应先判断它属于哪一类，再决定回答方式、输出形式和优先级。

### A. 技术研发类

典型主题：

- 管道数字化
- APS / AWP / workpacks
- 排产优化
- 三维模型特征分析
- 强化学习、启发式算法
- 焊接机器人、自动焊、视觉学习
- 研发路线设计与技术方案取舍

AI 应如何响应：

- 优先围绕用户的长期技术主线理解问题，而不是把它当成零散技术点
- 回答时同时考虑技术可行性、场景适配性、成本投入和落地路径
- 尽量给出阶段性路线，而不是只给终局方案
- 允许从最小可行实验、数据准备、验证指标、工具选型逐步展开

AI 需要避免：

- 把问题泛化成普通软件开发问题
- 忽视行业场景和真实施工环境
- 给出脱离数据条件、团队能力和项目节奏的“理想方案”

### B. 管理创新类

典型主题：

- 管理体系改革
- 组织协同
- 数字化转型
- 制度建设
- 季度研发管理计划
- 技术中心研发课题管理
- 物资、人力、固定资产、创新管理
- 技术中心运作机制
- 资源配置与职责划分

AI 应如何响应：

- 既要理解管理目标，也要理解技术和业务底层逻辑
- 优先帮助用户把抽象管理问题拆成机制、流程、责任、节奏和指标
- 输出应便于用户拿去执行、传达或试运行
- 需要同时考虑组织阻力、跨部门协同和现实可落地性

AI 需要避免：

- 给空泛的管理学语言
- 只讲“应该如何”，不讲“怎么推进”
- 忽视公司当前组织结构和用户正在扩大的管理职责

#### B1. 研发管理计划子场景

- 既要体现长期技术主线，也要落到季度目标、关键课题、验证场景和可检查产出

AI 需要避免：

- 把研发计划写成空泛年度总结
- 忽略管道数字化这个核心锚点

- 默认追求“有内容、有判断、有技术含量”的表达，而不是堆字数
- 输出前要尽量统一用户风格：务实、克制、聚焦成果、不过度注水

AI 需要避免：

- 套话、空话、评审腔

#### C1. 风格校准子场景

典型主题：

- 读取用户自己写的论文、提案、政策文件
- 读取用户对别人稿件的修改意见
- 对比“原稿 / AI 润色稿 / 用户终稿”
- 持续判断哪些是场景结构，哪些才是用户个人语言风格

AI 应如何响应：

- 默认把它当成持续采样任务，而不是一次性定稿任务
- 先说明这批材料更适合判断哪一层：场景结构、修改标准、个人风格、AI 润色偏差
- 允许把结论先收敛成“当前较稳”“待确认”“证据不足”三档
- 鼓励把不同材料继续细分为论文类、提案类、投标视频类、PPT 类等子场景

AI 需要避免：

- 把文体模板误判成用户个人偏好
- 因为看了少量材料就写死风格结论
- 只给抽象评价，不保留证据出处和场景说明

### D. 学习思考类

典型主题：

- 方法论提升
- 新技术学习
- 认知升级
- 路线判断
- 行业趋势分析
- 创业方向思考
- 思路梳理与长期规划

AI 应如何响应：

- 不只回答“是什么”，还要帮助用户搭框架、找变量、建判断标准
- 可以引导对比不同路径，但要尽量收敛到与用户主线最相关的部分
- 适合输出结构化框架、思维地图、路线图和推演逻辑
- 应帮助用户把抽象思考转回可行动的下一步

AI 需要避免：

- 只做知识搬运
- 提供与用户主线无关的大而全分析
- 把讨论停留在概念层面，不回到用户自己的场景

## 混合型需求识别

用户的真实需求经常不是单一类别，而是复合型问题。后续 AI 默认要检查是否同时包含以下维度：

- 技术 + 管理
- 技术 + 汇报
- 管理 + 表达
- 技术 + 管理 + 产业化
- 学习思考 + 路线决策

如果是混合型需求：

- 不要只按单一类别回答
- 先指出主类别和次类别
- 输出内容优先围绕主类别组织，但保留次类别的关键约束

## 回应优先级规则

当用户的问题跨多个类别时，后续 AI 默认按下列顺序判断重点：

1. 先识别这是不是用户长期主线课题
2. 再识别这是要“落地执行”还是“整理表达”
3. 再识别是否涉及组织推动或资源配置

## 三层记忆原则

为了避免只有结论、没有依据，后续 AI 应默认把重要信息拆成 3 层：

### 1. 结论层

文件：

- `summary.md`

适合放：

- 已经比较稳定的判断
- 当前对用户、项目、目标的总结
- 可直接复用的约束、偏好和决策

### 2. 证据层

文件：

- `evidence.md`

适合放：

- 用户的关键原话
- 能支撑某个结论的具体描述
- 用户自己打字时流露出的语气、表达风格和细节
- 有助于后续检查是否存在矛盾、变化或更细化补充的依据

原则：

- 不需要保存所有原话，只保留那些对长期理解用户有价值的原话
- 尤其保留：角色定义、长期目标、关键经历、强烈偏好、反感点、重要判断

### 3. 问题层

文件：

- `open_questions.md`

适合放：

- 还没有完全确认的信息
- 未来值得继续追问的问题
- 当前结论里仍有模糊地带、边界问题或潜在矛盾

原则：

- 后续 AI 不要假装都已经知道
- 应通过持续追问和校准，把问题层逐步转化为结论层

### /commit 工作流

## `/commit` 是什么

`/commit` 不是“保存聊天记录”。

它的真正含义是：

- 把当前主题段里值得长期保留的增量，沉淀回用户的记忆系统
- 更新需要更新的主题主档案
- 让未来的 agent、模型和用户自己，都能复用这次新增的 context

## 默认单位：一个主题一个 commit

`/commit` 默认处理的不是一问一答，也不是整库历史，而是：

- 当前聊天线程中
- 自上一个清晰分界点以来
- 围绕同一个主题连续展开的这一整段协作内容

所以默认原则是：

- 一个主题段，一个 commit

这里的“主题段”通常意味着：

- 经过多轮来回讨论
- 已经形成比较稳定的结论、证据、问题或规则
- 用户认为这段内容已经值得沉淀

## 什么不算一个独立 commit 单位

- 单条消息
- 单轮问答
- 整个仓库所有历史对话

如果一个聊天线程中明显切换到了另一个主题，默认应视为下一次 commit 的候选，而不是继续混进上一个主题段。

## `/commit` 什么时候应该触发

适合触发的情况：

- 形成了新的稳定结论
- 明确了新的长期目标、偏好、工作方法或判断标准
- 产生了值得保留的关键原话
- 某个固定主题需要更新
- 出现了新的长期主题，值得新建到 `memory/domains/`
- 当前主题段明显降低了未来重复解释成本

不一定要触发的情况：

- 只是普通来回讨论，没有形成稳定增量
- 只是临时试错，没有长期价值
- 只是执行性聊天，没有新的方法、结论或主题更新
- 只是重复已有内容，没有新增信息

## 不要按“每天一次”机械 commit

`/commit` 不应该按日期触发，而应该按“有没有高价值增量”触发。

所以：

- 今天聊了很多，但没有新的稳定结论，可以不 commit
- 今天只聊了一小段，但形成了重要判断，就值得 commit

## 连续工作时怎么用

如果今天和明天都在推进同一个主题：

- 不需要每天都重新开始
- 明天继续时，先加载已有 context，再接着聊
- 只有当当前主题段新增了值得长期保留的内容时，才触发 `/commit`

推荐心智模型：

- `load-context` 负责开始
- `/commit` 负责收尾

## 新项目时怎么用

如果是一个新的长期方向、新课题或新项目：

1. 先新建一条会话
2. 先在会话层沉淀内容
3. 如果发现这个主题会长期存在，再考虑升格为 `memory/domains/` 下的固定主题
4. 当前主题段形成稳定增量时再判断是否 `/commit`

也就是说：

- 新项目通常先开新会话
- 不一定一开始就新建主题层

## 跨 agent / 跨平台时怎么用

如果从 Codex 切到 Cursor、Claude Code 或其他 agent：

1. 新 agent 先读取：
   - `memory/snapshots/active_context.md`
   - 需要时再读相关 `memory/domains/*.md`
2. 开始新一轮协作
3. 当前主题段结束后，如果有高价值增量，再执行 `/commit`

关键点：

- 跨平台时最重要的是先加载 context
- `/commit` 负责把新增内容写回同一个 Git 仓库

## `/commit` 的标准动作

当用户说“commit 这次对话”时，后续 AI 默认把它理解为：

- “请把我们当前这个主题段里已经形成的新增内容沉淀下来”

然后执行以下动作：

1. 判断当前主题段是否真的有长期增量
2. 如果有，补全或更新当前会话中的：
   - `summary.md`
   - `evidence.md`
   - `open_questions.md`
   - `domain_updates.md`
3. 根据 `domain_updates.md` 决定是否更新：
   - `memory/profiles/`
   - `memory/domains/`
4. 重新生成：
   - `memory/snapshots/active_context.md`
5. 然后执行：
   - `git add .`
   - `git commit -m "..."`
   - `git push`

## `/commit` 前的判断问题

后续 AI 在触发 `/commit` 前，默认先判断这 4 个问题：

1. 当前主题段有没有新增的稳定结论？
2. 当前主题段有没有值得保留的关键原话？
3. 当前主题段有没有需要更新的固定主题？
4. 当前主题段有没有降低未来重复解释成本？

只要其中至少有 1 到 2 项明显成立，通常就值得 commit。

## `/commit` 的输出目标

一次好的 `/commit`，不只是“保存了内容”，而是应该让未来的使用者能更快进入状态：

- 未来的你自己
- 未来的 Codex
- 未来的 Claude Code
- 未来的 Cursor agent
- 未来的其他模型和自动化流程

### 场景 1：连续工作

- 继续工作
- 当前主题段形成稳定增量时再 `/commit`

### 场景 2：新项目

### 可追溯回答模板

## 目标

当后续 AI 引用用户的长期偏好、风格、目标或方法时，不应只给一个抽象结论，而应尽量做到可追溯、可解释、可回到原始证据。

## 默认回答结构

当用户问：

- “你为什么觉得我喜欢这个？”
- “你是根据什么判断的？”
- “这个偏好是怎么来的？”
- “你怎么知道我在这个场景下要求更高？”

后续 AI 默认按以下 4 层来回答：

### 1. 结论层

先明确回答当前判断是什么。

例如：

- 你确实很重视“结构有没有立住”
- 你对画面与配音的一致性要求很高
- 你不喜欢无效素材和重复信息

### 2. 来源层

指出这个判断主要来自哪里。

优先顺序：

1. `memory/domains/` 中的固定主题主档案
2. `memory/profiles/` 中的长期资料
3. 某次会话的 `summary.md`
4. 某次会话的 `evidence.md`
5. 某份导入资料，如 `memory/imports/`

### 3. 场景层

说明这个判断在什么场景下尤其明显。

例如：

- 在视频脚本场景里
- 在 PPT 修改场景里
- 在管理材料场景里
- 在技术成果表达场景里

### 4. 例子层

尽量给出一个具体例子，说明：

- 当时原本是什么
- 用户要求改成什么
- 背后的判断标准是什么

例如：

- 原来按人物分章节，用户要求改成按问题分章节
- 原来页面堆很多系统截图，用户要求换成更代表性的素材
- 原来画面与字幕不一致，用户要求按字幕含义重新选素材

## 推荐回答句式

后续 AI 可以优先使用这种结构：

- 结论：
- 来源：
- 场景：
- 例子：

如果用户只想要简短说明，可以压缩成 2 句：

- 先说结论
- 再补一句最关键的来源或例子

## 什么时候必须带来源

以下情况，后续 AI 应尽量带来源说明：

- 用户质疑判断依据
- 用户要求确认某项偏好是否真实稳定
- 用户想知道这个结论来自哪次对话或哪类材料
- 涉及写作风格、长期目标、价值判断、工作方法等高影响信息

## 什么时候可以只给结论

以下情况可以先给结论，再在用户追问时补来源：

- 用户只是快速推进当前任务
- 结论已经在当前对话里刚刚形成
- 用户更关心下一步怎么做，而不是结论来源

## 回答边界

- 不要把没有证据的印象说成稳定结论
- 如果证据不足，要明确说“这是当前推断，不是已确认事实”
- 如果结论主要来自某次单一会话，也应说明它的稳定性还需要后续观察

## 对当前系统的要求

为了支持这种回答方式，后续 AI 应持续维护：

- `domains` 中的稳定主题结论
- `conversations` 中的证据层
- `imports` 中的资料文本来源

这样未来在回答“为什么这么判断”时，就能从主题层一路追到证据层。

### Context 哲学

## 核心判断

- 一个没有用户 context 的强模型，和一个路边随机遇到的聪明人，本质上没有太大区别，因为它并不了解用户正在做什么，也无法真正进入协作状态。
- 人与 AI 是否能高质量 co-work，关键不只在模型是否聪明，更在于双方是否共享足够高质量的 context。
- 未来 AI 时代真正重要的基础设施之一，是把个人或公司的业务、背景、目标、偏好和历史上下文建模好，让 AI 可以持续在其上工作。

## 为什么要用 Git 管理 context

- Git 天然支持跨设备、跨 agent、跨模型迁移
- Git 有云端同步能力，可以把 context 变成可携带资产，而不是绑定在某个平台的单次会话里
- Git 的 `commit` 机制本身带有仪式感，能把“沉淀 context”变成一次明确的贡献动作
- Git 让用户可以主动控制哪些内容值得长期保留，哪些只是临时对话

## 用户的方法论

- 不是所有内容都要保存，重点是保存“值得沉淀”的内容
- 真正高价值的不是聊天记录本身，而是那些能降低后续协作摩擦的 context
- context 系统存在的意义，是让用户在任何时候都能带着自己的背景，去测试最新的 agent、模型和框架，而不是被动受限于某个平台

## 一个重要认知

- 用户看到的，不等于模型看到的
- context 的重要性就在于：它是用户知道自己看到了什么，也知道模型能看到什么的共同区域
- 如果不主动管理共同可见的 context，就会出现用户以为自己说清楚了，但模型其实没有在同一共享背景上工作的情况

## 关于 skill 的看法

- 默认不应盲目给模型叠加 skill 或过多预设约束
- 更好的顺序是先让模型直接尝试，再根据真实瓶颈决定是否需要 skill、规则或额外框架
- skill 像攻略，能提高稳定下限，但也可能限制探索上限
- 因此是否使用 skill，应该是基于任务需要，而不是默认动作

## 这套哲学对当前系统的启发

- 记忆系统要服务于“降低人与 AI 的摩擦力”
- 记忆系统既要跨会话，也要跨模型、跨 agent、跨设备
- 记忆系统不能只存结论，还要保留一部分原话证据和待确认问题
- 记忆系统不能只堆历史记录，还要形成固定主题主档案
- 每次重要对话后，都应判断这次内容是否值得进入长期主题层

## 当前适合坚持的做法

- 只提交高价值 context，而不是无脑全量提交
- 对重要对话使用 `commit` 思维，保留人为确认感
- 优先构建可迁移的 context 系统，再去比较模型优劣
- 先让模型裸跑，再决定是否补 skill 或更强约束

## 当前不建议走得太快的方向

- 不建议一开始就无时无刻自动同步所有内容
- 不建议过早把高度私密的情绪材料与工作 context 混在一起
- 不建议把系统复杂化到压倒实际使用体验

## 对未来产品化的含义

- 个人 context 管理系统，未来可以演化为个人 AI 操作系统
- 同样的思想也可以扩展到公司级 context 管理，即把公司业务、流程、知识、目标和约束建模后，再让 AI 在上面工作
- 从这个角度看，未来很多 AI SaaS 的核心，可能并不是模型本身，而是 context 建模能力
